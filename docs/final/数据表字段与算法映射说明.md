# 《数据表字段与算法映射说明》（字段中文说明增强版）

> 适用范围：减脂微信小程序当前项目基线
> 目标：把“算法用到哪些字段、字段落在哪张表、字段之间如何流转”讲清楚，方便产品、前后端、Cursor 后续统一理解。
> 依据：当前基线 PRD、主 PRD、基线对齐 SQL。

---

## 1. 文档目的

这份文档不重复讲公式本身，而是专门回答一个问题：

**每个核心算法，具体依赖哪些表字段，计算结果又写回哪些表字段。**

也就是把下面三件事对应起来：

- 算法输入字段
- 计算过程
- 落库输出字段。

为了避免后续开发、联调、测试、AI 辅助开发时出现理解偏差，本文档进一步补充：

1. 每张核心表的业务职责
2. 每个关键字段的**中文名称**与**业务含义**
3. 字段在算法中的角色：是输入、过程值、结果值，还是仅用于展示/关联
4. 多张表之间的字段流转逻辑。

---

## 2. 总体映射关系

当前系统的字段流转主线可以概括为：

**用户档案表 → 预算表 → 记录表 → 汇总表 → 统计展示**

对应现行表如下：

- 用户身份主表：`lw_user`
- 用户档案表：`user_profile`
- 用户预算表：`user_budget_config`
- 食物主数据表：`food`
- 食物分类表：`food_category`
- 餐次头表：`meal_record`
- 饮食明细表：`diet_record`
- 运动项目表：`sport_item`
- 运动记录表：`sport_record`
- 体重记录表：`user_weight_record`
- 日汇总表：`daily_summary`
- 识图任务表：`meal_photo_recognition`。

其中：

- `user_profile` 负责保存预算输入条件
- `user_budget_config` 负责保存预算算法结果
- `diet_record` / `sport_record` 负责保存每天的原始行为数据
- `daily_summary` 负责保存日维度聚合结果。

进一步从业务层面理解：

- **身份层**解决“这个数据是谁的”
- **档案层**解决“用户当前身体与目标状态是什么”
- **预算层**解决“系统算出来应该怎么吃”
- **行为记录层**解决“用户实际上做了什么”
- **汇总层**解决“这一天最终结果是什么”
- **统计展示层**解决“按周、按趋势如何呈现”。

## 3. 用户身份与档案字段映射

## 3.1 `lw_user`：身份层字段

`lw_user` 是当前统一后的用户主表，核心作用是保存微信登录身份和账号级信息。主要字段包括：

- `id`
- `openid`
- `unionid`
- `nickname`
- `avatar`
- `phone`
- `status`
- `register_source`
- `created_at`
- `updated_at`。

### 3.1.1 字段中文说明


| 字段名               | 中文名称      | 说明                        |
| ----------------- | --------- | ------------------------- |
| `id`              | 用户ID      | 用户主键，所有业务表通过该字段关联用户       |
| `openid`          | 微信OpenID  | 微信小程序用户唯一标识，同一小程序下唯一      |
| `unionid`         | 微信UnionID | 微信开放平台统一标识，可用于跨应用识别同一用户   |
| `nickname`        | 昵称        | 用户昵称，用于页面展示               |
| `avatar`          | 头像        | 用户头像地址                    |
| `phone`           | 手机号       | 用户绑定手机号                   |
| `status`          | 用户状态      | 用户当前状态，如正常、停用等            |
| `register_source` | 注册来源      | 用户来源渠道，如 `wx_miniprogram` |
| `created_at`      | 创建时间      | 用户记录创建时间                  |
| `updated_at`      | 更新时间      | 用户记录最后更新时间                |


### 3.1.2 算法角色说明

这张表本身**不直接参与预算公式**，它主要提供：

- 用户唯一主键 `id`
- 微信身份标识 `openid`
- 登录绑定手机号
- 个人展示基础资料。

因此在算法层面，`lw_user` 更像所有业务表的“归属主键来源”，不是预算和统计的直接输入表。
换句话说：

- 它决定“是谁”
- 不决定“怎么算”。

## 3.2 `user_profile`：档案输入层字段

`user_profile` 是预算算法的核心输入表，主要字段包括：

- `user_id`
- `gender`
- `age`
- `height_cm`
- `initial_weight_kg`
- `current_weight_kg`
- `target_weight_kg`
- `target_date`
- `profile_completed`。

### 3.2.1 字段中文说明


| 字段名                 | 中文名称     | 说明                            |
| ------------------- | -------- | ----------------------------- |
| `user_id`           | 用户ID     | 对应 `lw_user.id`，表示是哪位用户的档案    |
| `gender`            | 性别       | 用于区分男女 BMR 公式，通常 1=男，2=女，0=未知 |
| `age`               | 年龄       | 用户年龄，参与基础代谢计算                 |
| `height_cm`         | 身高（厘米）   | 用户身高，参与基础代谢计算                 |
| `initial_weight_kg` | 初始体重（kg） | 用户开始减脂时的起始体重                  |
| `current_weight_kg` | 当前体重（kg） | 用户当前最新体重快照                    |
| `target_weight_kg`  | 目标体重（kg） | 用户希望达到的体重                     |
| `target_date`       | 目标达成日期   | 用户期望达到目标体重的日期                 |
| `profile_completed` | 是否完成建档   | 标识用户是否完成资料完善                  |


### 3.2.2 这些字段与算法映射关系如下

- `gender`：用于区分男女 BMR 公式
- `age`：用于 BMR 公式
- `height_cm`：用于 BMR 公式
- `current_weight_kg`：用于 BMR、TDEE、待减体重计算
- `target_weight_kg`：用于待减体重、总热量差、每日缺口计算
- `target_date`：用于剩余天数和每日推荐缺口计算
- `initial_weight_kg`：不参与预算公式，只用于累计减重和趋势展示。

所以可以把 `user_profile` 理解为：
**预算输入字段表 + 趋势起点字段表。**

### 3.2.3 关键业务理解补充

这里最容易混淆的是三个体重字段的职责：

1. `initial_weight_kg`：减脂起点，只用于“从哪里开始”
2. `current_weight_kg`：当前快照，用于预算与当前状态展示
3. `target_weight_kg`：目标终点，用于计算还要减多少。

因此：

- 修改 `initial_weight_kg`，通常影响累计减重和趋势展示
- 修改 `current_weight_kg`，会直接影响预算
- 修改 `target_weight_kg`，会影响总热量差与每日推荐缺口。

## 4. 预算表字段映射

## 4.1 `user_budget_config`：预算输出层字段

`user_budget_config` 是预算结果表，核心字段包括：

- `user_id`
- `use_custom_bmr`
- `custom_bmr`
- `activity_level`
- `calculated_bmr`
- `calculated_tdee`
- `recommended_deficit`
- `daily_budget`
- `carb_target_g`
- `protein_target_g`
- `fat_target_g`
- `effective_date`。

### 4.1.1 字段中文说明


| 字段名                   | 中文名称        | 说明                      |
| --------------------- | ----------- | ----------------------- |
| `user_id`             | 用户ID        | 对应哪位用户的预算配置             |
| `use_custom_bmr`      | 是否使用自定义基础代谢 | 1=使用自定义BMR，0=使用系统计算值    |
| `custom_bmr`          | 自定义基础代谢     | 用户或系统人工指定的 BMR 值        |
| `activity_level`      | 活动系数        | 用于从 BMR 推导 TDEE 的活动倍数   |
| `calculated_bmr`      | 计算基础代谢      | 系统根据身高、体重、年龄、性别计算得到     |
| `calculated_tdee`     | 计算总日消耗      | 系统根据 BMR 和活动系数计算出的 TDEE |
| `recommended_deficit` | 推荐每日热量缺口    | 为实现目标体重建议每天制造的热量差       |
| `daily_budget`        | 每日摄入预算      | 用户每天建议摄入的总热量上限          |
| `carb_target_g`       | 碳水目标（g）     | 每日建议摄入的碳水目标克数           |
| `protein_target_g`    | 蛋白目标（g）     | 每日建议摄入的蛋白目标克数           |
| `fat_target_g`        | 脂肪目标（g）     | 每日建议摄入的脂肪目标克数           |
| `effective_date`      | 生效日期        | 本条预算配置从哪一天开始生效          |


字段与算法的对应关系如下。

### `activity_level`

这是 TDEE 计算因子，真正参与公式的是 decimal 值，不是文案。迁移逻辑中已给出历史映射，例如 1.200、1.375、1.550、1.725、1.900。

### `calculated_bmr`

对应 BMR 公式计算结果：

- 男：`10*体重 + 6.25*身高 - 5*年龄 + 5`
- 女：`10*体重 + 6.25*身高 - 5*年龄 - 161`。

### `calculated_tdee`

对应：

`TDEE = BMR × activity_level`。

### `recommended_deficit`

对应：

`recommended_deficit = (current_weight_kg - target_weight_kg) × 7700 ÷ 剩余天数`。

### `daily_budget`

对应：

`daily_budget = calculated_tdee - recommended_deficit`。

### `carb_target_g` / `protein_target_g` / `fat_target_g`

对应宏量目标换算：

- `carb_target_g = daily_budget × 50% ÷ 4`
- `protein_target_g = daily_budget × 20% ÷ 4`
- `fat_target_g = daily_budget × 30% ÷ 9`。

### `effective_date`

表示该预算配置从哪一天开始生效。它非常重要，因为预算不是永远只有一套，而是允许随着用户档案变化生成新版本。

### 4.1.2 字段职责补充说明

`user_budget_config` 的本质是：
**把档案输入数据转换成可直接使用的预算结果。**

也就是说：

- `user_profile` 是输入层
- `user_budget_config` 是结果层

后续首页、日报、统计页在需要使用预算时，优先读这张表，而不是每次从档案字段现算。这样可以保证：

1. 页面展示一致
2. 预算有版本概念
3. 历史数据可回溯。

## 5. 体重相关字段映射

## 5.1 `user_profile` 与 `user_weight_record` 的分工

体重字段在系统里分两层：

- `user_profile.current_weight_kg`：当前体重快照
- `user_weight_record.weight_kg`：按日期记录的体重流水。

`user_weight_record` 关键字段包括：

- `user_id`
- `record_date`
- `weight_kg`
- `source`
- `remark`。

### 5.1.1 字段中文说明


| 字段名           | 中文名称   | 说明                       |
| ------------- | ------ | ------------------------ |
| `user_id`     | 用户ID   | 对应哪位用户的体重记录              |
| `record_date` | 记录日期   | 这条体重记录对应的日期              |
| `weight_kg`   | 体重（kg） | 当天记录的实际体重值               |
| `source`      | 来源     | 体重记录来源，如 `manual/system` |
| `remark`      | 备注     | 体重记录备注信息                 |


这两张表的算法分工如下：

- 预算公式读取 `user_profile.current_weight_kg`
- 趋势图优先读取 `user_weight_record`
- 累计减重使用 `user_profile.initial_weight_kg` 与 `user_profile.current_weight_kg`
- 每次新增体重记录后，应同步刷新 `user_profile.current_weight_kg` 为最新值

这就是“历史明细 + 当前快照”的标准建模方式。

## 5.2 初始体重字段的算法定位

`user_profile.initial_weight_kg` 的作用是：

- 作为减脂起点
- 计算累计减重
- 提供计划页、趋势页、个人资料页的展示基线。

它**不映射到** BMR、TDEE、daily_budget 等预算结果字段。
所以从字段映射角度看，它属于“展示算法输入字段”，不属于“预算算法输入字段”。

### 5.2.1 关键口径补充

这里后续开发必须统一：

- `current_weight_kg` 变了 → 预算可能要重算
- `initial_weight_kg` 变了 → 预算通常不重算，只影响展示口径

这是非常关键的字段边界。

## 6.2 `food`

`food` 是饮食换算的主数据表，关键字段与算法映射如下：

- `id`：供 `diet_record.food_id` 关联
- `category_id`：分类归属
- `name`：食物名称
- `image`：食物图片
- `gi_level`：GI 级别展示
- `calories_per_100g`：按克计算热量的基础值
- `calories_per_unit`：按单位计算热量的基础值
- `unit_name`：单位名称
- `standard_weight_g`：标准单位重量，辅助单位转克重
- `edible_portion_rate`：可食部比例
- `carb_per_100g`
- `protein_per_100g`
- `fat_per_100g`
- `keywords`
- `is_custom`
- `status`。

### 6.2.1 字段中文说明


| 字段名                   | 中文名称      | 说明                    |
| --------------------- | --------- | --------------------- |
| `id`                  | 食物ID      | 食物主键                  |
| `creator_user_id`     | 创建人用户ID   | 若是自定义食物，记录由谁创建        |
| `category_id`         | 分类ID      | 对应 `food_category.id` |
| `name`                | 食物名称      | 食物展示名称                |
| `image`               | 食物图片      | 食物图片地址                |
| `gi_level`            | GI等级      | 低GI / 中GI / 高GI 等展示字段 |
| `calories_per_100g`   | 每100克热量   | 按克计算的基础热量值            |
| `calories_per_unit`   | 每单位热量     | 按单位计算的基础热量值           |
| `unit_name`           | 单位名称      | 如个、片、碗、杯等             |
| `standard_weight_g`   | 标准单位重量（g） | 一个单位大约等于多少克           |
| `edible_portion_rate` | 可食部比例     | 食物实际可食部分占比            |
| `carb_per_100g`       | 每100克碳水   | 按克换算碳水的基础值            |
| `protein_per_100g`    | 每100克蛋白   | 按克换算蛋白的基础值            |
| `fat_per_100g`        | 每100克脂肪   | 按克换算脂肪的基础值            |
| `keywords`            | 搜索关键词     | 搜索联想、模糊匹配使用           |
| `is_custom`           | 是否自定义     | 1=用户自定义食物，0=系统食物      |
| `status`              | 状态        | 是否启用                  |


其中直接参与计算的核心字段是：

- `calories_per_100g`
- `calories_per_unit`
- `carb_per_100g`
- `protein_per_100g`
- `fat_per_100g`。

映射关系为：

### 按重量计算

- `diet_record.calories_total ← food.calories_per_100g × weight_g ÷ 100`
- `diet_record.carb_total_g ← food.carb_per_100g × weight_g ÷ 100`
- `diet_record.protein_total_g ← food.protein_per_100g × weight_g ÷ 100`
- `diet_record.fat_total_g ← food.fat_per_100g × weight_g ÷ 100`。

### 按单位计算

- `diet_record.calories_total ← food.calories_per_unit × amount`

如果系统进一步按单位折克重，则可通过 `standard_weight_g` 辅助换算，但当前表结构本身已经允许直接按单位热量落值。

### 6.2.2 字段理解补充

`food` 这张表是**饮食计算的模板层**，而不是行为层。
也就是说：

- 它定义“这个食物本身是什么”
- 不定义“用户今天吃了多少”

真正的行为记录在 `diet_record`。这个边界一定要保持清晰。

## 7. 餐次与饮食明细字段映射

- `id`

- `record_date`
- `meal_type`
- `total_calories`
- `total_carb_g`
- `total_protein_g`
- `total_fat_g`
- `food_count`

### 7.1.1 字段中文说明


| 字段名               | 中文名称   | 说明                |
| ----------------- | ------ | ----------------- |
| `id`              | 餐次ID   | 一餐的主键             |
| `user_id`         | 用户ID   | 该餐属于哪位用户          |
| `record_date`     | 记录日期   | 该餐属于哪一天           |
| `meal_type`       | 餐次类型   | 早餐 / 午餐 / 晚餐 / 加餐 |

- `food_id`
- `food_name_snapshot`
- `image_snapshot`
- `gi_level_snapshot`
- `amount`
- `amount_unit`
- `weight_g`
- `calories_total`
- `carb_total_g`
- `protein_total_g`
- `fat_total_g`
- `source`

## 12. 关键字段映射表（精简版）

下面给你一版最常用的“字段 → 算法 → 输出”速查。

### 12.1 档案到预算

- `user_profile.gender/age/height_cm/current_weight_kg` → BMR → `user_budget_config.calculated_bmr`
- `user_budget_config.calculated_bmr + activity_level` → TDEE → `user_budget_config.calculated_tdee`
- `user_profile.current_weight_kg/target_weight_kg/target_date` → 推荐缺口 → `user_budget_config.recommended_deficit`
- `calculated_tdee - recommended_deficit` → 每日预算 → `user_budget_config.daily_budget`
- `daily_budget` → 三大目标 → `carb_target_g/protein_target_g/fat_target_g`。

### 12.2 食物到饮食明细

- `food.calories_per_100g + diet_record.weight_g` → `diet_record.calories_total`
- `food.carb_per_100g + diet_record.weight_g` → `diet_record.carb_total_g`
- `food.protein_per_100g + diet_record.weight_g` → `diet_record.protein_total_g`
- `food.fat_per_100g + diet_record.weight_g` → `diet_record.fat_total_g`。

### 12.3 饮食明细到餐次头

- `Σ diet_record.calories_total` → `meal_record.total_calories`
- `Σ diet_record.carb_total_g` → `meal_record.total_carb_g`
- `Σ diet_record.protein_total_g` → `meal_record.total_protein_g`
- `Σ diet_record.fat_total_g` → `meal_record.total_fat_g`
- `count(diet_record)` → `meal_record.food_count`。

### 12.4 饮食与运动到日汇总

- `Σ diet_record.calories_total` → `daily_summary.intake_calories`
- `Σ sport_record.calories_burned` → `daily_summary.exercise_calories`
- `daily_budget - intake + exercise` → `daily_summary.remain_calories`
- `daily_budget - intake + exercise` → `daily_summary.budget_deficit_calories`
- `TDEE + exercise - intake` → `daily_summary.actual_deficit_calories`
- `min/max(diet_record.record_time)` → `first_meal_time/last_meal_time`
- `末餐 - 首餐` → `eating_window_hours`。

### 12.5 体重与趋势

- `user_weight_record.weight_kg` → 趋势图纵轴
- `user_weight_record.record_date` → 趋势图横轴
- `user_profile.current_weight_kg` → 当前展示体重
- `user_profile.initial_weight_kg - user_profile.current_weight_kg` → 累计减重。

## 13. 当前项目后续必须统一的字段理解

后续开发时，字段层面最容易跑偏的地方有这些：

### 13.1 `current_weight_kg` 与 `weight_kg`

- `user_profile.current_weight_kg` 是快照
- `user_weight_record.weight_kg` 是历史流水。

### 13.2 `daily_budget` 与 `remain_calories`

- `daily_budget` 是预算目标
- `remain_calories` 是当天实时剩余额度。

### 13.3 `recognized_calories` 与 `confirmed_calories`

- 前者是识图识别值
- 后者是用户确认值
- 只有确认写入后才成为正式饮食统计来源。

### 13.4 `food` 与 `diet_record`

- `food` 是主数据
- `diet_record` 是行为数据真源。

### 13.5 `sport_item` 与 `sport_record`

- `sport_item` 是项目模板
- `sport_record` 是实际消耗真源。

### 13.6 `meal_record` 与 `diet_record`

- `meal_record` 表示“一餐”
- `diet_record` 表示“一餐里的每一个食物明细”

因此：

- 餐次展示、餐次评价更多依赖 `meal_record`
- 摄入计算与汇总真源依赖 `diet_record`。

### 13.7 `recommended_*` 与 `confirmed_*`

在识图链路里：

- `recommended_*` 是系统建议
- `confirmed_*` 是用户确认

后续任何统计或正式入库逻辑，必须以确认后的结果为准。

## 14. 统一结论

这套系统的字段映射规律，最终可以归纳为一句话：

**档案表提供预算输入，预算表保存预算结果，记录表沉淀用户每天的真实行为，汇总表承接统计结果。**

也就是说：

- `user_profile` 决定怎么算预算
- `user_budget_config` 保存预算答案
- `diet_record` / `sport_record` 保存每天发生了什么
- `daily_summary` 保存这一天最终算出来什么
- `user_weight_record` 保存体重趋势真源
- `meal_photo_recognition` 保存识图中间态和确认映射关系。

进一步说：

- **用户身份层**解决“是谁”
- **档案层**解决“当前身体与目标是什么”
- **预算层**解决“系统建议怎么吃”
- **行为层**解决“用户实际做了什么”
- **汇总层**解决“这一天最后结果是什么”
- **统计层**解决“按周、按趋势怎么解释”

因此，后续无论是前端页面、后端接口、SQL 脚本，还是 Cursor 自动生成逻辑，都必须遵守本文档的字段语义与映射关系，不能再出现：

- 用模板表当行为表
- 用展示字段当计算字段
- 用识别值当确认值
- 用快照字段替代历史真源字段。
