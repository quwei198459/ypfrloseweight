## 1. 执行总原则

### 1.1 最小改动原则

- 不重构现有登录体系。
- 不重构现有认证鉴权。
- 不重构全局响应结构。
- 不重构全局异常处理。
- 不重构现有首页整体结构。
- 不重构现有客服体系。
- 不借本需求重写减肥业务相关逻辑。

### 1.2 安全原则

- 宜远智能应用 ID、密钥只允许放在后端安全配置中。
- DeepSeek API Key 只允许放在后端安全配置中。
- 前端不得保存、拼接、透传第三方密钥。
- 日志不得打印完整 `Authorization`、宜远密钥、DeepSeek Key。
- 用户面部图片属于敏感信息，后台查看和 PDF 导出必须鉴权。

### 1.3 业务隔离原则

- 皮肤检测手机号白名单和赠送次数必须与减肥业务分开页面管理。
- 皮肤检测次数不得复用减肥次数。
- 减肥白名单逻辑不得因本需求被破坏。

### 1.4 固定检测项原则

本次只调用已购买的 6 项：

| 项目 | 字段 | detect_type |
|---|---|---:|
| 斑点 | `spot` | 2 |
| 毛孔 | `pore` | 4 |
| 肤质 | `skin_type` | 8 |
| 痤疮 | `acne` | 16 |
| 皱纹 | `wrinkle` | 8192 |
| 黑头 | `blackhead` | 32768 |

合计：

```text
detect_types = 40990
```

禁止调用：黑眼圈、肌肤年龄、水分、肤色、敏感度、三庭五眼等未购买项目。

### 1.5 小程序 UI 一致原则

- 页面必须沿用当前小程序已有风格、规范和颜色体系。
- 截图只作为流程、模块、内容和布局层级参考。
- 如果截图风格与现有小程序冲突，以现有小程序规范为准。
- 不允许以“优化”“美化”为由做成割裂的新视觉系统。

### 3.4 阶段 0 摸底结论

#### 3.4.1 小程序首页

现有首页文件：`frontend/src/pages/home/index.vue`。

页面结构：

- 顶部品牌区。
- 搜索栏。
- 热量统计卡片。
- 早餐/午餐/晚餐/加餐/运动快捷入口。
- 减脂计划 Banner。

皮肤检测入口建议插入位置：

- 优先插入在热量统计卡片之后、减脂计划 Banner 之前。
- 原因：需求截图入口位于首页中部服务选择区域；该位置不破坏顶部核心热量模块，也不影响现有减脂计划 Banner。
- 实现时只新增一个入口卡片/服务卡片，不重排现有首页模块。

视觉约束：

- 复用首页现有浅绿色背景、圆角卡片、按钮、阴影、间距规范。
- 不引入独立蓝紫色或医美风格整套视觉。

#### 3.4.2 小程序请求、拍照与上传能力

现有请求封装：

- `frontend/src/utils/http.ts`
- 支持 `httpGet`、`httpGetAuth`、`httpPost`、`httpPostAuth`、`httpDeleteAuth`。
- 现有接口统一解析后端 `ApiResponse<T>`，`code=0` 为成功。

现有拍照能力：

- 页面：`frontend/src/pages/photograph/index.vue`
- 流程封装：`frontend/src/composables/usePhotographFlow.ts`
- 已支持：
  - `uni.chooseImage` 相册。
  - `uni.chooseImage` 相机。
  - 微信小程序相机权限申请。
  - 内嵌拍照失败时降级系统相机。
  - `readLocalFileAsBase64` 本地图片转 base64。
  - 上传/识别/失败/成功阶段状态。

皮肤检测建议：

- 复用 `readLocalFileAsBase64`。
- 可参考 `photograph/index.vue` 的相机授权、相册/相机降级、加载状态处理。
- 不建议直接复用餐前拍一拍的业务 composable，因为餐前拍一拍包含餐别、食物识别、比例调节、饮食记录写入等专属逻辑。
- 皮肤检测应新增独立页面和独立 composable/API，避免误伤餐前拍一拍。

#### 3.4.3 用户资料来源

现有用户 store：

- `frontend/src/stores/user.ts`
- `frontend/src/stores/userProfile.ts`

现有用户接口：

- `GET /api/v1/user/profile`
- `POST /api/v1/user/profile/update`

现有用户资料字段：

- 昵称。
- 头像。
- 性别。
- 年龄。
- 身高。
- 当前体重。
- 目标体重。
- 目标日期。
- 手机号。

皮肤检测所需字段对比：

- 姓名：现有可用 `nickname` 自动带入，但应允许用户单独修改为检测姓名。
- 性别：现有可用 `gender` 自动带入。
- 出生日期：现有只有 `age`，没有 `birthday`。
- 居住地：现有用户资料没有 `residence`。

建议：

- 皮肤检测资料页从现有用户资料自动带入昵称、性别、年龄。
- 皮肤检测提交接口单独接收 `name`、`birthday`、`gender`、`residence`。
- 不建议为了皮肤检测直接改造全局用户资料表，除非产品确认出生日期和居住地要成为全局资料字段。

#### 3.4.5 后端白名单、次数、鉴权与文件能力

现有白名单/次数能力：

- 表：`photo_recognition_member_phone`
- 表：`photo_recognition_quota_log`

- 管理接口：`AdminPhotoRecognitionMemberController`

现有逻辑特点：

- 按手机号白名单判断是否可用。
- 有总次数、已使用次数、剩余次数。
- 支持后台调整次数。
- 支持使用明细日志。

关键限制：

- 该能力属于餐前拍一拍/拍照识别，不能直接与皮肤检测混用。
- 皮肤检测需要独立页面管理，建议阶段 1 新增独立表：`skin_detection_whitelist`。
- 可以参考现有 `PhotoRecognitionQuotaService` 的结构实现独立 `SkinDetectionQuotaService`。

现有鉴权：

- 小程序端使用 `AuthUserResolver` 解析 Bearer Token。
- 后台端使用 `AdminAuthResolver`。
- 皮肤检测用户端和后台端均可复用现有鉴权方式，不改全局鉴权。

现有文件能力：

- `AvatarStorageService` 可保存 Base64 或 multipart 图片到指定目录。
- 当前命名偏头像，但能力可复用。
- 皮肤检测原图和结果图建议后续新增更语义化的存储服务或在现有服务上最小扩展，避免把皮肤检测图片混入头像目录语义。

### 4.6 阶段 1 产出

#### 4.6.1 新增数据库迁移脚本

新增脚本：`database/migrations/V026__skin_detection.sql`。

新增表：

1. `skin_detection_whitelist`
   - 用途：皮肤检测手机号白名单与次数。
   - 关键字段：
     - `phone`：手机号，唯一索引。
     - `status`：1 启用，0 停用。
     - `total_times`：总赠送/调整次数，默认 3。
     - `used_times`：已使用次数，默认 0。
     - `quota_updated_at`：最近一次次数调整时间。
   - 索引：手机号唯一索引、状态索引、创建时间索引。

2. `skin_detection_record`
   - 用途：皮肤检测记录主表。
   - 关键字段：
     - `user_id`：用户 ID，关联 `lw_user.id`。
     - `phone`、`user_name`、`gender`、`birthday`、`age`、`residence`：检测时用户资料快照。
     - `image_url`：用户原始检测图片。
     - `third_party_id`：宜远返回标识，如有。
     - `detect_types`：默认 40990。
     - `overall_score`：综合分。
     - `ai_summary`、`ai_summary_json`：DeepSeek 综合分析。
     - `third_party_raw`：宜远原始返回 JSON。
     - `status`：pending/success/failed。
     - `fail_reason`、`deepseek_status`、`deepseek_error`：失败与 AI 状态。
   - 索引：用户、手机号、状态、创建时间、综合分。

3. `skin_detection_item`
   - 用途：皮肤检测 6 项单项明细。
   - 关键字段：
     - `record_id`：检测记录 ID。
     - `item_code`：spot/pore/skin_type/wrinkle/blackhead/acne。
     - `item_name`：中文名。
     - `score`：单项分。
     - `level`：程度/等级。
     - `result_text`：结果说明。
     - `result_image`：结果图。
     - `metrics_json`：数量、面积占比、分类等扩展指标。
     - `sort_order`：展示顺序。
   - 约束：同一记录下 `record_id + item_code` 唯一。

4. `skin_detection_quota_log`
   - 用途：皮肤检测次数调整与使用明细。
   - 关键字段：
     - `whitelist_id`：皮肤检测白名单 ID。
     - `phone`：手机号快照。
     - `user_id`：使用人或操作人用户 ID。
     - `record_id`：检测记录 ID。
     - `change_type`：grant/reduce/consume。
     - `change_count`：变化次数。
     - `before_total_times`、`before_used_times`、`after_total_times`、`after_used_times`：变化前后快照。
     - `operator_type`、`operator_name`、`remark`：操作信息。

影响说明：

- 只新增表，不修改现有表。
- 不影响减肥业务表。
- 不影响餐前拍一拍 `photo_recognition_*` 表。

## 12. 当前风险清单

| 风险 | 说明 | 应对 |
|---|---|---|
| 痤疮接口可能无 `score` | 会影响 6 项平均分 | 联调确认；必要时产品确认换算规则 |
| 第三方图片 3 个月过期 | 历史报告可能图片失效 | 检测成功后考虑转存结果图 |
| DeepSeek 返回格式不稳定 | 报告展示可能异常 | 后端做 JSON 解析和文本兜底 |
| 皮肤检测次数与减肥次数混淆 | 用户权益错误 | 独立页面、独立接口、独立数据表优先 |
| 小程序页面风格割裂 | 用户体验不一致 | 强制沿用现有小程序颜色和组件规范 |
| 密钥泄露 | 安全事故 | 后端配置、日志脱敏、禁止前端写入 |
