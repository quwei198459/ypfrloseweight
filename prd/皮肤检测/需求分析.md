# 皮肤检测需求分析

> 依据：`需求简要.md`、`pic/` 下页面截图、OCR 识别结果、宜远智能测肤 API 文档。  
> OCR 输出目录：`prd/皮肤检测/ocr/`。  
> 安全说明：原需求中包含宜远接口密钥与 DeepSeek Key，本文档不记录明文密钥；实现时必须放入后端环境变量或安全配置，禁止写入前端与 Git。

## 1. 需求目标

本次需求是在小程序中新增“皮肤检测/AI 测肤”业务闭环：用户从首页入口进入检测流程，填写或确认个人资料，按提示拍照上传，后端调用宜远智能测肤 API 获取 6 项检测结果，再调用 DeepSeek 生成综合分析，最终在小程序展示综合报告、详细报告，并通过“定制护肤方案/联系客服”引导用户添加客服微信，服务于美容产品销售转化。

成功标准：

- 首页指定位置新增皮肤检测入口。
- 用户可按截图流程完成检测。
- 后端完成宜远智能 6 项已购买 API 调用。
- 用户检测结果完整入库。
- 6 项检测分计算平均值作为综合分。
- 综合分析由 DeepSeek 根据 6 项结果生成。
- 皮肤检测手机号白名单与赠送次数需和减肥业务分开页面管理。
- 皮肤检测白名单默认赠送 3 次，管理员可调整。
- 后台可查询检测记录并导出 PDF。
- 点击“定制护肤方案/定制方案/联系客服”展示后台客服信息。

```text

  ↓

  ↓
勾选/确认皮肤检测协议
  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓
综合报告
  ↓
查看详细报告 / 点击定制护肤方案联系客服
```

关键规则：

- 进入或开始检测前应查询剩余皮肤检测次数。
- 皮肤检测次数与减肥业务次数完全分开。
- 剩余次数不足时不能提交检测。
- 第三方检测成功后扣减 1 次。
- 图片质量不合格、框脸失败、第三方服务异常时建议不扣次数。
- 检测中应防止重复提交。
- 检测成功但 DeepSeek 分析失败时，基础检测报告仍可保存，综合分析用兜底文案。

## 5. 第三方测肤 API 分析

API 服务地址：

```text
https://api.yimei.ai
```

鉴权方式：

```text
Authorization: Basic base64(应用ID:密钥)
```

本次使用正脸测肤接口：

```text
POST /v2/api/face/analysis/${detect_types}
```

请求图片字段：

- `image`：可为文件或图片 URL。
- 建议小程序先上传到本项目后端，由后端以 `multipart/form-data` 调用第三方。

结果图片访问：

```text
https://api.yimei.ai/fileSvr/get/${filename}
```

重要限制：第三方图片服务器只保留 3 个月。如后台历史报告需要长期可见，建议将第三方结果图转存到本项目文件服务。

### 5.1 本次确认使用的 6 项 detect_types

| 序号 | 购买项目名称 | 第三方字段 | detect_type 值 | 说明 |
|---:|---|---|---:|---|
| 1 | 斑点 | `spot` | 2 | 返回斑点数量、分数、分类、结果图等 |
| 2 | 毛孔 | `pore` | 4 | 返回毛孔数量、分数、面积占比、结果图等 |
| 3 | 肤质 | `skin_type` | 8 | 返回肤质类型、分数/区域分类等 |
| 4 | 痤疮 | `acne` | 16 | 返回痤疮数量、等级、结果图等 |
| 5 | 皱纹 | `wrinkle` | 8192 | 返回皱纹数量、分数、分类、结果图等 |
| 6 | 黑头 | `blackhead` | 32768 | 返回黑头数量、分数、面积占比、结果图等 |

本次合计：

```text
detect_types = 2 + 4 + 8 + 16 + 8192 + 32768 = 40990
```

请求地址：

```text
POST /v2/api/face/analysis/40990
```

### 5.2 不纳入本次调用的项目

虽然截图明细中出现或 API 文档支持以下项目，但本次未购买，不应调用：

- 黑眼圈 `dark_circle=131072`。
- 肌肤年龄 `age=1`。
- 色素/棕色斑 `chloasma=64`。
- 粗糙度 `roughness=128`。
- 肤色 `color=256`。
- 水分 `moisture=4096`。
- 红区、敏感度、三庭五眼等其它项目。

### 5.3 分数字段注意

已购买 6 项中，大多数文档返回 `score`。但 `acne` 在文档中主要返回 `count`、`level`、`filename`，是否稳定返回 `score` 需联调确认。

推荐规则：

- 如第三方返回 `score`，直接使用。
- 如 `acne` 不返回 `score`，需后端根据 `level/count` 做换算，或产品确认默认评分规则。
- 换算规则不能前端临时写死，应由后端统一处理。

## 6. DeepSeek 综合分析

需求：将 6 项检测结果一起传给 DeepSeek，让其扮演皮肤专家生成综合结论。

建议输入：

- 用户年龄、性别、居住地等非敏感基础信息。
- 6 项检测名称、分数、等级、关键指标。
- 单项结果简述。

不建议传入：

- 手机号。
- 明文姓名。
- 用户原始图片。
- 第三方完整原始 JSON 中无关字段。

建议输出结构：

```json
{
  "summary": "综合结论",
  "mainProblems": ["主要问题"],
  "careAdvice": ["护理建议"],
  "productDirection": ["产品方向"],
  "riskReminder": "风险提醒"
}
```

失败兜底：

- DeepSeek 失败不应导致第三方检测记录丢失。
- 报告可展示兜底文案：“综合分析暂不可用，请联系客服获取详细方案”。
- 后台记录失败原因，便于重试。

## 7. 分数规则

- 本次固定 6 项：斑点、毛孔、肤质、皱纹、黑头、痤疮。
- 如果第三方返回 `score`，直接作为单项分。
- 如果痤疮等项目无 `score`，后端需按产品确认的规则换算为 0-100 分。
- 综合分由后端计算：

```text
综合分 = round((斑点分 + 毛孔分 + 肤质分 + 皱纹分 + 黑头分 + 痤疮分) / 6)
```

建议：

- 后端统一计算并存储综合分。
- 前端只展示后端结果，避免前后不一致。
- 如果核心检测项失败，建议整次检测失败并不扣次数。

建议表名：`skin_detection_record`

核心字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| user_id | 用户 ID |
| phone | 手机号 |
| user_name | 用户姓名/昵称 |
| gender | 性别 |
| birthday | 出生日期 |
| age | 年龄，可由出生日期计算 |
| residence | 居住地 |
| image_url | 用户原始检测图片 |
| third_party_id | 宜远返回 id |
| detect_types | 固定为本次 40990，或保存实际调用值 |
| overall_score | 综合分 |
| ai_summary | DeepSeek 综合分析文本 |
| ai_summary_json | DeepSeek 结构化结果 |

| status | pending/success/failed |
| fail_reason | 失败原因 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 8.2 检测明细表

建议表名：`skin_detection_item`

核心字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| record_id | 检测记录 ID |
| item_code | `spot`/`pore`/`skin_type`/`wrinkle`/`blackhead`/`acne` |
| item_name | 中文名称 |

### 8.3 皮肤检测白名单和次数

需求明确：皮肤检测的手机号白名单和赠送次数要和减肥的分开页面管理。

推荐新增独立表或独立业务模块：`skin_detection_whitelist`。

核心字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| phone | 手机号 |
| total_times | 总赠送次数，默认 3 |
| used_times | 已使用次数 |
| remaining_times | 剩余次数 |
| remark | 备注 |
| enabled | 是否启用 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

如果为了最小改动复用现有白名单表，也必须在后台菜单、页面、查询条件和业务逻辑上与减肥白名单分开，避免两类次数互相影响。

## 12. 安全要求

- 宜远应用 ID、密钥、DeepSeek Key 只允许后端保存。
- 不写入前端 `.env`、页面代码或 Git。
- 后端日志不得打印完整密钥和完整 Authorization。
- 用户面部图片属于敏感信息，后台访问和 PDF 导出必须鉴权。
- 第三方结果图若需长期保存，应转存到本项目文件服务。

建议配置项：

```text
YIMEI_API_BASE_URL=https://api.yimei.ai
YIMEI_CLIENT_ID=后端安全配置
YIMEI_CLIENT_SECRET=后端安全配置
YIMEI_FACE_DETECT_TYPES=40990
DEEPSEEK_API_KEY=后端安全配置
SKIN_DETECT_DEFAULT_TIMES=3
```

## 13. 风险与待确认

| 问题 | 风险 | 建议 |
|---|---|---|
| 痤疮接口是否返回 score 未确认 | 影响 6 项平均分 | 联调确认；如无 score，由产品确认换算规则 |
| 截图出现黑眼圈明细但未购买 | 误调用未授权接口 | 本次不实现黑眼圈，只复用样式 |
| 第三方图片 3 个月过期 | 历史报告图失效 | 成功后转存结果图 |
| DeepSeek 返回格式不稳定 | 报告展示异常 | 后端做 JSON 解析兜底 |
| 重复提交 | 重复扣次数/重复计费 | 后端加幂等和进行中状态 |
| 皮肤检测次数与减肥次数混淆 | 用户权益错误 | 独立后台页面和独立数据模型 |
| 密钥泄露 | 安全事故 | 环境变量化，日志脱敏 |

## 14. 验收标准

### 用户端

- 首页指定位置可进入皮肤检测。
- 皮肤检测相关页面沿用当前小程序已有风格、颜色体系、组件规范和交互习惯，不出现独立割裂的新视觉风格。
- 检测介绍页包含三步流程和协议确认。
- 资料页可自动带入并修改姓名、出生日期、性别、居住地。
- 有次数用户可完成检测，无次数用户被拦截。
- 可拍照、重拍、提交检测。
- 检测中有动画效果。
- 检测成功后展示综合报告。
- 综合分等于 6 项平均分。
- 明细项只包含斑点、毛孔、肤质、皱纹、黑头、痤疮。
- 点击定制护肤方案展示后台客服信息。

### 后端

- 第三方 API 由后端调用。
- 固定调用 `/v2/api/face/analysis/40990` 或等价配置。
- 不调用未购买项目。
- 密钥不暴露给前端。
- 成功检测结果完整入库。
- 成功检测扣减 1 次皮肤检测次数。
- 图片质量/框脸失败不扣次数。
- DeepSeek 失败不影响基础检测报告保存。
- 接口具备用户鉴权与记录归属校验。

### 管理后台

- 有独立的皮肤检测白名单/次数管理页面。
- 不与减肥白名单页面混用。
- 新增手机号默认赠送 3 次。
- 管理员可调整皮肤检测次数。
- 可查询检测记录。
- 可查看检测详情。
- 可导出 PDF。

## 15. 推荐开发顺序

1. 后端用 6 项 `detect_types=40990` 完成宜远 API 调用 PoC。
2. 联调确认痤疮是否返回 `score`。

4. 实现皮肤检测次数校验与扣减。
5. 实现检测记录保存和报告查询。
6. 接入 DeepSeek 综合分析。
7. 实现小程序首页入口和检测流程页面。
8. 实现综合报告、详细报告和客服弹窗。
9. 实现后台独立白名单管理、记录查询和 PDF 导出。
10. 按截图逐项做 UI 高保真修复与验收。
